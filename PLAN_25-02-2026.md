# Plan de Implementación — Sprint 2
**Fecha:** 25-02-2026

## Resumen de decisiones

| Ítem | Decisión |
|---|---|
| Tipos evento | STI: base `RecurringEvent` + `SocialEvent` / `TallerEvent` / `CongresoEvent` |
| Foto | Upload de archivo → servida como estático desde la API |
| Backoffice | Sección admin dentro del Ionic app (rutas protegidas por rol `admin`) |
| Geolocalización | `lat`/`lng` en el Venue → botón "¿Cómo llegar?" abre Google Maps |
| Aforo | Campo `aforoMaximo` en Venue, definido por admin |
| Balance | Renombrado a % follower/leader — resumido en tarjeta, detallado en analíticas |
| % nivel | Solo en panel de analíticas |
| Corazón | Solo cambio de icono: el tick de "Voy" pasa a ser un corazón |

---

## Fase 1 — Backend: modelo de datos

- [ ] **1.1 Venue entity** — añadir `latitud: number`, `longitud: number`, `aforoMaximo: number`
- [ ] **1.2 RecurringEvent — refactor a STI**
  ```
  RecurringEvent (base, @TableInheritance)
  ├── SocialEvent    → sin campos extra
  ├── TallerEvent    → precio: number, instructor: string
  └── CongresoEvent  → precio: number, fechaFin: string
  ```
  Añadir `fotoUrl: string | null` en la base.
- [ ] **1.3 File upload** — instalar `multer` + `@types/multer`, endpoint `/uploads` para servir fotos
- [ ] **1.4 Shared types** — actualizar `WeeklyEvent`, `Venue`, `EventAnalytics` con los nuevos campos

## Fase 2 — Backend: API

- [ ] **2.1 Admin role** — añadir `admin` al enum `Rol`. Guard `AdminGuard` que rechaza si `rol !== admin`
- [ ] **2.2 Events API**
  - `GET /api/events/week?tipo=social|taller|congreso` — filtro opcional por tipo
  - `POST /api/events/:id/photo` — upload de foto (multipart/form-data), solo admin
  - CRUD completo: `POST /api/events`, `PATCH /api/events/:id`, `DELETE /api/events/:id`
- [ ] **2.3 Venues API** — CRUD completo con `latitud`, `longitud`, `aforoMaximo`
- [ ] **2.4 Analytics** — `PredictionService` actualizado:
  - `getRoleBalance()` devuelve porcentajes numéricos `{ followers: 60, leaders: 40 }`
  - `getNivelDistribution()` devuelve % por nivel
  - `getAmbiente()` usa `aforoMaximo` del venue en lugar de umbrales fijos

## Fase 3 — Frontend mobile: mejoras UX

- [ ] **3.1 EventCard** — foto de cabecera, chip de tipo, `% followers / % leaders`, icono corazón en "Voy"
- [ ] **3.2 Home/Agenda** — chips de filtro por tipo encima de la lista
- [ ] **3.3 Analytics panel** — barra visual `% followers / % leaders` + desglose de niveles en %
- [ ] **3.4 Event detail** — botón "¿Cómo llegar?" → Google Maps + aforo del venue

## Fase 4 — Backoffice (admin)

- [ ] **4.1 AdminGuard** — redirige a `/tabs/home` si `rol !== admin`
- [ ] **4.2 `/admin/events`** — lista + crear/editar/borrar eventos + upload de foto
- [ ] **4.3 `/admin/venues`** — lista + crear/editar locales (lat/lng/aforo)
- [ ] **4.4 `/admin`** — dashboard con stats globales

## Orden de ejecución

```
Fase 1 (modelo) → Fase 2 (API) → Fase 3 (mobile) → Fase 4 (backoffice)
```
